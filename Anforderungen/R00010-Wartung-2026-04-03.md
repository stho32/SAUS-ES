### Verfuegbare globale Architekturen

| Architektur | Global | Lokal | Passt zu SAUS-ES? |
|-------------|--------|-------|-------------------|
| dotnet-avalonia-clean-architecture | Ja | Nein | Nein (anderer Stack) |

| svelte-app-architecture | Ja | Nein | Nein (anderer Stack) |

### Ergebnis

Die Architektur `laravel-blade-tailwind` ist die naechste Uebereinstimmung:
- Gleiche Sprache (PHP 8.4)
- Gleicher Use-Case (CRUD-lastige Anwendung mit oeffentlichen und geschuetzten Bereichen)
- Gleiche DB-Technologie (MySQL)

**Abweichungen:**
- SAUS-ES nutzt kein Framework — Laravel-Architektur setzt MVC, Eloquent, Blade, Middleware voraus
- SAUS-ES nutzt Bootstrap via statische Dateien — Laravel-Architektur nutzt Tailwind CSS
- SAUS-ES hat keine Tests — Laravel-Architektur definiert vollstaendige Test-Pyramide mit Pest

Eine Migration auf die `laravel-blade-tailwind`-Architektur wuerde einen kompletten Rewrite bedeuten, waere aber langfristig die richtige Entscheidung fuer Wartbarkeit, Sicherheit und Testbarkeit.

## Phase 4: Verbesserungsvorschlaege

### Vorschlag 1: CSRF-Schutz implementieren

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Mittel

**Ist-Zustand:** Kein CSRF-Schutz implementiert. Alle 23 API-Endpunkte und Formulare akzeptieren POST-Requests ohne Token-Validierung. CLAUDE.md und README.md behaupten faelschlicherweise, CSRF-Schutz sei vorhanden.

**Soll-Zustand:** Alle state-changing Requests (POST/PUT/DELETE) muessen ein CSRF-Token validieren.

**Umsetzung:**
1. Token-Generierung in `php/includes/auth.php`: `$_SESSION['csrf_token'] = bin2hex(random_bytes(32))`
2. Token als Hidden-Field in alle Formulare einbetten
3. Token-Validierung als Funktion `validateCsrfToken()` in allen API-Endpunkten
4. SameSite=Strict fuer Session-Cookie (bereits in .htaccess, aber auch in PHP setzen)
5. CLAUDE.md und README.md korrigieren

### Vorschlag 2: Rate-Limiting einfuehren

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Mittel

**Ist-Zustand:** Kein Rate-Limiting. Master-Link-Codes koennen per Brute-Force erraten werden. API-Endpunkte sind ungeschuetzt gegen Flooding.

**Soll-Zustand:** IP-basiertes Rate-Limiting fuer Auth-Endpunkte und API-Calls.

**Umsetzung:**
1. Rate-Limiting-Tabelle in MySQL: `rate_limits (ip, endpoint, attempts, window_start)`
2. Funktion `checkRateLimit(string $ip, string $endpoint, int $maxAttempts, int $windowSeconds): bool`
3. In `auth.php` und allen API-Endpunkten einbinden
4. Empfohlen: 10 Versuche/Minute fuer Auth, 60 Requests/Minute fuer APIs

### Vorschlag 5: Content-Security-Policy Header hinzufuegen

**Kategorie:** Sicherheit
**Prioritaet:** Mittel
**Aufwand:** Klein

**Ist-Zustand:** .htaccess setzt X-Frame-Options, X-Content-Type-Options und HSTS, aber kein Content-Security-Policy (CSP).

**Soll-Zustand:** CSP-Header verhindert Inline-Script-Injection und beschraenkt externe Ressourcen.

**Umsetzung:**
In `.htaccess` nach Zeile 14 hinzufuegen:
```apache
Header set Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'"
```
Hinweis: `unsafe-inline` ist noetig solange Inline-Scripts/Styles genutzt werden. Langfristig auf externe Dateien umstellen.

### Vorschlag 13: XSS in JavaScript-Kontext beheben

**Kategorie:** Sicherheit
**Prioritaet:** Mittel
**Aufwand:** Klein

**Ist-Zustand:** In `php/news.php:324` wird `htmlspecialchars(addslashes($news['title']))` fuer JavaScript-Kontext verwendet. `addslashes()` ist nicht sicher fuer JS-Strings.

**Soll-Zustand:** `json_encode()` fuer JavaScript-Kontext verwenden.

**Umsetzung:**

```php
// Alt:
onclick="deleteNews(<?= $news['id'] ?>, '<?= htmlspecialchars(addslashes($news['title'])) ?>')"
// Neu:
onclick="deleteNews(<?= $news['id'] ?>, <?= htmlspecialchars(json_encode($news['title'])) ?>)"
```

### Strategische Einordnung

SAUS-ES ist ein funktionales Nischenprodukt fuer Wohnungsbaugenossenschaften mit einer klaren Daseinsberechtigung. Die Hauptstaerke liegt in der Einfachheit (kein Login, Master-Links) und dem spezialisierten Feature-Set (Voting, Partner-Links, oeffentliche Ansicht).

**Investitionsentscheidung:** Weitere Investitionen lohnen sich, wenn:
- Die Software aktiv genutzt wird und Nutzer-Feedback einfliessen
- Die kritischen Sicherheitsluecken (CSRF, Rate-Limiting) zeitnah geschlossen werden
- Mindestens eine Basis-Testabdeckung aufgebaut wird

**Risiken bei Nicht-Handeln:**
- CSRF-Luecke koennte bei Bekanntwerden ausgenutzt werden
- Ohne Tests steigt das Risiko von Regressionen bei jeder Aenderung
- Ohne CI/CD ist die Code-Qualitaet von manueller Disziplin abhaengig

**Langfrist-Perspektive:** Eine schrittweise Migration auf Laravel wuerde die meisten Sicherheits- und Tooling-Probleme strukturell loesen und die Weiterentwicklung deutlich beschleunigen. Die vorhandene globale Architektur `laravel-blade-tailwind` bietet eine direkt nutzbare Vorlage.
